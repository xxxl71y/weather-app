> 从零到上线 Android 天气应用的完整开发历程。适用于个人练手项目、H5 套壳 App、免费 API 驱动的小产品。

## 三、做得好的

### 1. 免费 API 优先
选择 Open-Meteo（无需密钥、无配额限制）让项目零成本起步。不用注册开发者账号、不用担心欠费。

### 2. 渐进式提交
每次只改一个功能点，commit message 写清动机。后期排查编译失败时能快速定位。

### 3. 固定签名密钥
提交 keystore 到仓库，CI 统一使用。彻底解决了"每次更新要卸载重装"的问题。

### 4. 自审查流程
后期建立了"改完必查"的纪律——每个改动发布前用 agent 扫描一遍 Java 编译、JS 逻辑、文本一致性。减少了 CI 编译失败次数。

### 5. 缓存优先的加载策略
localStorage + SharedPreferences 双向缓存，App 重开即显天气数据。用户感知的加载时间从 5-15 秒降到亚毫秒级。

## 四、做得不好的

### 1. 零测试覆盖
没有单元测试、没有 UI 测试、没有 CI 编译门禁。每次发布靠人工"看看有没有报错"。导致多次"发布→CI 构建失败→修→再发"的循环。

**以后应该：** CI 加 gradle compileDebug 检查，至少保证编译通过。

### 2. 单文件过千行
index.html 超 1000 行，包含天气逻辑、设置管理、时间选择器、缓存读写。改一个功能要在文件里翻很久。

**以后应该：** 300 行触发拆分。JS/CSS/HTML 分离。按功能模块组织。

### 3. 版本号分散
版本号散落在 build.gradle.kts、index.html APP_VERSION、README、Release 描述四处。每次发版要手动对齐，经常遗忘。

**以后应该：** 单一版本源（build.gradle 或 package.json），其他位置构建时注入。

### 4. 架构选型随意
选用 WebView 套壳是为了"一套 HTML 跨平台"，但后期 JS ↔ Java 桥接维护成本持续累积，且 Android 审核对纯 WebView 应用不友好。

**以后应该：** 先评估原生方案 vs 跨平台方案的成本曲线。练手项目可以容忍，但要知道代价。

### 5. 文案反复修改
"监测→预测"、"全平台→多端"、"8 点→自由调节"等文案改动拖了很多轮，每次改动都是一个新的 commit。

**以后应该：** 发布前列出用户可见文案 checklist，一次定稿。

### 6. 中间产物大量浪费
Windows 端做了 PyInstaller 打包、Inno Setup 安装器、反病毒问题排查，最后全删。iOS 配置也是。

**以后应该：** 先用最小代价验证一个平台的核心价值，确认需要再扩。不要一开始就想"全平台"。

### 7. 前台服务走了弯路
一开始用 Foreground Service 保活，发现通知栏残留问题后又推翻回到 WorkManager。两次架构变更。

**以后应该：** 设计时优先考虑 Android 系统限制（通知必须可见、后台有限制），选择最保守可靠的方案。

## 五、关键技术决策复盘

| 决策 | 当时理由 | 现在评价 |
|------|---------|---------|
| WebView 而非原生 UI | 一套 HTML 跨平台 | 练手 OK，生产环境不推荐 |
| WorkManager 而非前台 Service | 通知栏不能有残留 | ✅ 正确 |
| 双向缓存 (localStorage + SharedPreferences) | 快速加载 | ✅ 效果显著 |
| 自定义时间选择器 | 避 AM/PM 问题 | ⚠️ 能用但代码量大 |
| JS ↔ Java 桥接 | 网页设置同步到后台 | 必需但脆弱，无类型安全 |

## 六、发布前 Checklist

- [ ] 所有文本扫描：版本号、名称、功能描述一致
- [ ] 本地 gradle compileDebug 通过
- [ ] 核心路径手动测一遍：安装→打开→设置→通知→关闭→重开
- [ ] README 与当前功能匹配
- [ ] Release 描述不过时
- [ ] 确认没有残留的删除功能引用（import、配置、文案）

## 七、技术栈速览

| 层级 | 技术 | 用途 |
|------|------|------|
| 天气数据 | Open-Meteo API | 免费、无密钥、全球 |
| 地理编码 | BigDataCloud | 坐标→中文地址 |
| IP 定位 | ipapi.co | GPS 失败时的后备 |
| Android 壳 | WebView + Java | 加载 HTML、权限、通知 |
| 后台任务 | WorkManager | 周期天气检测、早晚定时推送 |
| 缓存 | localStorage + SharedPreferences | 天气数据秒开 |
| 桥接 | @JavascriptInterface | JS ↔ Java 数据同步 |
| 图标 | Vector Drawable (XML) | 自适应图标、无分辨率问题 |
| CI/CD | GitHub Actions | Release 触发 APK 构建 |
| 分发 | GitHub Releases | APK 直链下载 |

## 八、给下一个项目的建议

1. **先做减法**：列出不做的事，比列出要做的事更重要
2. **API 选型三问**：免费？无密钥？稳定？三个都是 yes 才用
3. **单文件 < 300 行**：超过就拆，不妥协
4. **版本号只在一处定义**：其他地方构建时注入
5. **CI 门禁放编译检查**：不通过不合并
6. **文案一次定稿**：不反复改用户可见文字
7. **记录踩坑清单**：每次发布遇到的问题记下来，积累自己的知识库
